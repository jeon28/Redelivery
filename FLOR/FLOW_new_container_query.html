FLOR 신규 컨테이너 조회 흐름

2026-05-21 · 처음 보는 컨테이너 (예: DFSU9999999) 1건 조회 시 단계별 순서

이 문서는 무엇?
"가장 단순한 케이스" — Florens 에 처음 등록되는 신규 컨테이너 1건 — 의 조회 흐름을 코드 기준으로 정리. precleared / Closed / VOID 같은 분기는 §부록에 별도 표기.

1. 한눈에 보는 흐름

[입력 검증] │ _is_flor_container() — DFSU/FSCU prefix 통과해야 진행 ▼ [세션 복원] ~5s │ storage_state 로드 또는 슬라이더 로그인 ▼ [Phase 1: Status 탭] ~10~15s │ _navigate_to_status │ └─ "Redelivery Status" 탭 클릭 │ └─ _select_search_by_unit_no — Unit No. 모드 강제 │ _search_one(c) │ └─ _fill_unit_input (3단계 캐스케이드) │ └─ Search 버튼 JS 클릭 │ └─ 결과 파싱 │ └─ rows=0 → "발급된 반납번호 없음 (신청 필요)" │ _status_row_is_new_candidate → True │ └─ new_candidates 리스트에 추가 ▼ [Phase 2: Apply 위저드] ~15s │ _navigate_to_apply │ Step 1 ─ Customer ID / Port / Depot │ Step 2 ─ Unit Numbers textarea │ Step 3 ─ _parse_step3 (3-패스 분류) │ └─ 신규 컨테이너 → 3차 pass: valid (신규 발급 후보) ▼ [Phase B: 자동 Confirm] ~5~10s │ _confirm_redelivery_order │ └─ Confirm 버튼 is-disabled 체크 │ └─ 클릭 전 PPR 셋 스냅샷 │ └─ 클릭 → 컴펌 다이얼로그 닫기 │ └─ 클릭 후 PPR 차집합 → 신규 PPR 식별 │ └─ unit ↔ PPR 매핑 (윈도우 우선 → 잔여 → 공유) ▼ [최종 응답] ~1s │ _precleared 정리 + 3상태 status 도출 ▼ UI: ✅ 가능 + 반납지 + 신규 PPR<번호>

2. 단계별 상세

0 입력 검증 + 세션 복원 ~5초

  1. 컨테이너 형식 검증 (_is_flor_container)
    • 정규식 ^[A-Z]{4}\d{6,7}$ + prefix 가 DFSU 또는 FSCU 인지
    • 실패 시 위저드 호출 자체 건너뛰고 → 🔢 넘버오류
  2. 세션 복원
    • storage_state 파일(/data/flor_session_HA.json 또는 SK.json) 로드 시도
    • 유효하면 곧장 /func/redelivery#/ 진입
    • 만료 시 슬라이더 캡차 + Sign-in 버튼 클릭 후 새 세션 저장

1 Phase 1 — Status 탭 우선 조회 ~10~15초

1-1. 탭 진입 — _navigate_to_status

  1. "Redelivery Status" 탭 클릭
  2. wait_for_load_state('networkidle', 10s) + sleep(1.5)
  3. Unit Number input attached 대기 (최대 10초)
  4. _select_search_by_unit_no() — Search By 라디오/탭 중 "Unit No." 텍스트 매칭 후 클릭. 이미 선택이면 skip.
    이 단계가 핵심. Status 탭 기본값이 'Redelivery No.' 모드일 때 Unit Number 입력란이 0×0 hidden 상태가 됨 → 이 모드 전환 없이는 검색이 빈 query 로 들어가 rows=0.

1-2. 컨테이너 검색 — _search_one(container)

  1. _fill_unit_input — 3단계 캐스케이드, 단계마다 input_value() 검증:
    1. Playwright fill() — visible idx 의 input 대상 (가장 안정)
    2. wrapper .el-input click + press_sequentially(delay=50) + Tab blur
    3. JS native setter + InputEvent(data, inputType) + blur — 최후
  2. 입력값 검증 로그 search input 검증 (visible): actual=..., expected=...
  3. Search 버튼 JS 클릭 — is-disabled 체크 후 evaluate 로 click
    • disabled 면 즉시 "Search 버튼 disabled (Vue 동기화 실패)" 에러 행 반환
  4. wait_for_load_state('networkidle', 20s)
  5. 결과 출현 대기 — wait_for_function 으로 tbody 행 또는 "no data/results" 텍스트까지 (최대 8초)
  6. 진단 로그 search diag: {url, rows, first_row, counter}
  7. 결과 헤더/행 파싱 (parse_cells 헬퍼)

1-3. 결과 판정

케이스처리
신규 (rows=0) reason = "발급된 반납번호 없음 (신청 필요)" → _status_row_is_new_candidate=True → new_candidates 추가
Open채택 → 가능 + EXPIRY DATE
Closed채택 → 🏁 반납완료 + OFF HIRE DATE
모두 VOID"재신청 가능" → 신규 후보로 분류
신규 컨테이너는 "발급된 반납번호 없음" 으로 분류 → Phase 2 로 이동.

2 Phase 2 — Apply 위저드 사전 검증 ~15초

2-1. Apply 탭 진입 — _navigate_to_apply

  1. page.goto(REDELIV_URL) + sleep 2s
  2. "Apply Redelivery" 탭 클릭 (이미 활성이어도 안전)

2-2. Step 1 — 설정 입력

  1. _select_customer_id(cust_id) — SINOKOR(SK) / HALINE(HA) 라디오 클릭. 이미 선택이면 skip.
  2. _select_port(port_substr) — Element UI 드롭다운 클릭 → 옵션 텍스트 substring 매칭
    • BUSAN → "PUSAN" (옛 표기)
    • INCHON → "INCHON"
    • GWANGYANG → "GWANGYANG"
  3. _select_depot(depot_str) — 두번째 드롭다운, depot 코드((KRPUS03) 등) 부분일치
  4. _step_next() — "Next" 버튼 → networkidle + sleep 1.5s

2-3. Step 2 — 컨테이너 입력

  1. _step2_fill_units(new_candidates) — textarea 에 \n 구분으로 일괄 입력
  2. _step_next() — "Next" 클릭

2-4. Step 3 — 사전 검증 결과 파싱 (_parse_step3)

모든 테이블 행을 수집한 뒤 3-패스 분류:

패스매칭분류
1차같은 행 안에 unit + reason 동시 존재invalid (over caps 등 즉시 거부)
2차reason-only 행 ↔ unit-only 행 — greedy nearest-neighbor 1:1 (row index 거리 최소)invalid + split_matched 로그 (precleared 등 분리행 케이스)
3차1·2차 모두 매칭 안 된 unit-only 행valid (신규 발급 후보)
신규 컨테이너는 일반적으로 3차 valid 로 분류 → Phase B 로 이동.

B Phase B — 자동 Confirm 실 발급 ~5~10초

주의: 이 단계 = 실 PPR 발급. memory project_flor_flow: "Confirm 클릭 = 즉시 실 발급 (PPR<5자리> Open, Pending 단계 없음)".

B-1. 대상 필터링

new_valid = [
    c for c in new_candidates
    if results[c].available             # Step3 valid
    and not results[c].booking_ref      # 아직 미발급
    and not results[c]._precleared      # precleared 가 아님
]

B-2. Confirm 실행 — _confirm_redelivery_order(new_valid)

  1. "Confirm Redelivery Order" 버튼 찾기 (fallback "Confirm" 도 시도)
  2. is-disabled / disabled 체크 → disabled 면 안전 가드로 스킵
  3. 클릭 전 페이지 텍스트 dump → pprs_before 셋 저장
  4. 버튼 click → networkidle + sleep 3s
  5. 컴펌 다이얼로그 (OK/Confirm/Yes/확인) 자동 닫기
  6. 클릭 후 페이지 텍스트 dump → 모든 PPR 추출 → pprs_before 와 차집합 = 신규 PPR 리스트

B-3. unit → PPR 매핑 (우선순위)

  1. 1차 — 페이지 텍스트에서 unit 번호 위치 주변 ±200/+400 문자 윈도우 안의 PPR 사용 (가장 정확)
  2. 2차 — 못 찾으면 잔여 PPR 을 unmapped unit 순서대로 1:1 할당
  3. 3차 — 그것도 없으면 첫 신규 PPR 공유 할당 (같은 PPR 묶음 케이스)

B-4. 결과 저장

results[unit].booking_ref = ppr
results[unit].depot = depot_str

4 최종 응답 정리 ~1초

  1. 누락 컨테이너 안전장치 (_error_row) — 어떤 경로로도 결과 못 채운 경우
  2. 3상태 status 일괄 도출:
    • completed / number_error 는 그대로 보존
    • available=True → status='available'
    • 그 외 → status='unavailable' + 기본 reason 채움
  3. 내부 가드 플래그 _precleared 제거 (API 응답에 노출 X)
  4. 입력 순서대로 list 반환 → 라우터 → 프론트

5. 부록 — 분기 케이스 흐름

5-1. precleared 일 때

  • Phase 1 Status 탭: rows=0 (precleared 는 Status 탭에 안 나타남)
  • Phase 2 Apply Step3: 1차 또는 2차 패스에서 invalid → _classify_reject §5-3 분기
  • 풀텍스트 reason 보존 + _precleared=True 가드
  • Phase B 스킵 (재발급 방지)
  • UI: 🟢 완료 + 파랑 배경 + 풀텍스트

5-2. Closed 일 때

  • Phase 1 Status 탭: rows≥1, status="Closed" 행 발견
  • _search_one 가 채택 → status='completed' + reason "OFF HIRE DATE …"
  • _status_row_is_new_candidate=False → 신규 후보 아님 → Phase 2/B 미발동
  • UI: 🏁 반납완료 + 회색 배경

5-3. Open (활성 PPR 보유)

  • Phase 1 Status 탭: rows≥1, status="Open" 행 발견 (또는 Voided 다수+Open 1)
  • _search_one 가 채택 → booking_ref=PPR… + close_date="EXPIRY DATE YYYY-MM-DD"
  • Phase 2/B 미발동
  • UI: ✅ 가능 + 녹색 배경 + EXPIRY DATE

5-4. 넘버오류 (타사 컨테이너 / 형식 오류)

  • _is_flor_container() 가 False 반환 → 위저드 호출 자체 건너뜀
  • 즉시 status='number_error' + 정해진 reason 부여
  • 다중 입력 시 정상 컨테이너는 별도로 위저드 진행 → 시간 절약
  • UI: 🔢 넘버오류 + amber 배경
