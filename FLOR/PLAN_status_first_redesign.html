FLOR 흐름 재설계: Status 우선 → Apply 폴백

같은 컨테이너 (DFSU7597484 / PPR78148) 에 대해 Apply Step 3 파싱이 한 번은 잡고 한 번은 못 잡는 불안정성을 두 차례 패치(bidirectional sweep + wait_for_function) 했지만 근본적 한계 잔존. 본 작업은 Apply 가 아니라 Redelivery Status 탭을 권위 있는 소스로 두는 재설계.

1. 현재 접근의 한계 정리

#원인현 대응
1 Element UI fixed column → 한 시각적 행이 두 개의 <table> 로 쪼개져 렌더링. 컨테이너 번호 / PPR 텍스트가 다른 row 에 분포. bidirectional sweep regex 추가 (커밋 f3a259d)
2 Unit Detail 영역이 Step 3 진입 후 가변 지연으로 그려짐. 같은 컨테이너인데 한번은 나오고 한번은 안 나옴. wait_for_function 으로 "Redelivery number:" 대기 (커밋 e9c4c1d)
3 Apply Redelivery 는 본질적으로 신규 신청 화면 — 이미 발급된 PPR 조회의 정식 진입점이 아님. 추가 패치를 해도 표면적 대응. 본 작업으로 해결

Redelivery Status 탭은 사이트가 명시한 PPR 상태 조회 화면이고, Redelivery No. / Status / Expiry Date / Depot Name 을 모두 단일 뷰에서 제공한다 (사용자 캡처 Image #5 확인). 이게 정답.

2. 새 흐름

  1. login (세션 재사용)
  2. Redelivery Status 탭 진입
  3. 각 컨테이너 Search by Unit No.
    • Open 행 발견 → ✅ 가능 + PPR + Expiry Date + Depot (Status row + Expand)
    • Closed/Void → 분류에 따라 completed 또는 unavailable
    • 결과 없음 → 신규 후보로 별도 보관
  4. 신규 후보가 있으면 Apply Redelivery 1회 실행 (현재 Step 1~3 흐름)
    • Step 3 valid → ✅ 가능 (booking_ref None — FLOR 사이트에서 Confirm 시 PPR 발급됨)
    • Step 3 invalid → ❌ + 사유 (over caps 등)

3. 코드 변경

위치변경
flor.py query() 본문 재구성 — Status 1단계 → Apply 2단계 (필요시만)
flor.py 신규 helper _status_lookup_one(container) — Status 탭에서 단건 조회 후 분류 (Open / Closed / Void / None 반환)
flor.py _parse_step3 제거 sweep fallback (_sweep_precleared) — Status 우선으로 더 이상 불필요
flor.py _parse_step3 wait_for_function 유지 (방어용 — Apply 폴백에서 혹시 precleared 행 보일 수 있음)
기존 helper 본체 변경 없음 — _search_one / _expand_and_extract / _navigate_to_status / _navigate_to_apply / Step1~3 헬퍼 그대로 활용

3.1 새 query() 골격

async def query(self, containers, region, depot=None):
    deduped = ...  # 기존과 동일
    results = {c: self._error_row(c, "조회 실패") for c in deduped}

    try:
        # === Phase 1: Status 탭 우선 조회 ===
        await self._navigate_to_status()
        new_candidates = []
        for c in deduped:
            try:
                row = await self._search_one(c)  # Open/Closed/Void/Empty 처리 포함
                if row.get("booking_ref") or row.get("close_date") or row.get("reason"):
                    # Status 에서 의미있는 정보 발견 → 채택
                    results[c].update(row)
                else:
                    # 결과 행이 없거나 빈 응답 → 신규 후보
                    new_candidates.append(c)
            except Exception as exc:
                logger.warning("FLOR %s status lookup 실패: %s", c, exc)
                new_candidates.append(c)

        # === Phase 2: 신규 후보들만 Apply 흐름 ===
        if new_candidates:
            depot_str = self._resolve_depot(region, depot)
            cust_id = COMPANY_CUSTOMER_ID.get(self.company)
            port_substr = PORT_OPTION_SUBSTR.get((region or "").upper())
            await self._navigate_to_apply()
            await self._select_customer_id(cust_id)
            await self._select_port(port_substr)
            await self._select_depot(depot_str)
            await self._step_next()
            await self._step2_fill_units(new_candidates)
            await self._step_next()
            await self._parse_step3(results, depot_str)

    except Exception as exc:
        logger.error("FLOR query error: %s\n%s", exc, traceback.format_exc())

    ...

3.2 신규 후보 판정 기준

_search_one 의 반환값 해석:

  • booking_ref 가 있고 status 가 Open → 정식 채택 (precleared)
  • reason 에 "재신청 가능 (이전 예약 모두 VOID)" 또는 "이미 반납됨" → 채택
  • 그 외 빈 행 / "발급된 반납번호 없음 (신청 필요)" → 신규 후보로 분류

위 분류는 _search_one 내부에서 이미 처리됨. query() 에서는 booking_ref / close_date / reason 중 하나라도 의미있는 값이면 채택, 아니면 신규 후보로 push.

4. 시간 영향

케이스현재 (Apply 우선)변경 후 (Status 우선)
precleared 1건~25s (Apply 15s + enrichment 6s + 파싱 불안정)~8s (Status 한 번만)
신규 1건~15s~22s (Status 6s + Apply 15s)
precleared 5 + 신규 5~45s~45s (Status 10×6s + Apply 1회 묶음)
precleared 10~80s (불안정)~60s
신규 10~15s~75s

신규 다건이 최악 — 6개 이상 경고문은 그대로 유효.

6. 검증

  1. DFSU7597484 1건 조회 → Status 경로로 즉시 ✅ 가능 + 반납번호 PPR78148 + 유효기간 2026-07-19. Apply 흐름 미발동.
  2. 신규 컨테이너 1건 조회 → Status 결과 없음 → Apply 흐름 → Step 3 valid → ✅ 가능 (booking_ref None).
  3. 의도적 invalid 케이스 (over caps 컨테이너) → Status 결과 없음 → Apply Step 3 invalid → ❌ + 사유.
  4. 혼합 5+5 (precleared + 신규) → 각각 올바른 경로로 분류.
  5. Status 탭 진입 실패 (네트워크) → query 흐름 전체 catch 에서 잡혀 사용자에게 detail 메시지 전달.
