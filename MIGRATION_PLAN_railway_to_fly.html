백엔드 이전 계획: Railway → Fly.io (NRT)

작성일 2026-05-21 · 즉시 이전 아님, 향후 결정 시 참조용.

이 문서의 위치: 결정 후 실행은 별도. 본 문서는 결정 직전 검토 + 작업자 체크리스트 용도. 실 작업 시 본 문서의 절차를 따라가며 단계별 사용자 승인.

0. 결정 배경

  • 오늘 (2026-05-20) Railway incident — free/trial/hobby plan 빌드 일시 정지 사고. 약 1시간 동안 배포 불가. Pro 는 영향 없음.
  • 이전에도 비슷한 흔적 — 커밋 로그에 chore: re-trigger after source restore, chore: trigger Railway build with current HEAD 등 수동 트리거 흔적 다수.
  • 현재 Railway 리전 SFO — 한국 사용자 / 홍콩 Florens 까지 RTT 모두 길음.
  • railway up 의 업로드 페이로드 약 777MB (frontend/node_modules + .next 가 함께 올라감) → timeout 빈발.

1. 결정 사항 — 백엔드만 이전

구성현재이후
프론트엔드 (Next.js)로컬 dev + Vercel 운영그대로 — 변경 없음
백엔드 (FastAPI + Playwright)Railway SFOFly.io NRT
볼륨 /data (49MB)Railway redelivery-volume 500MBFly volume 1GB
프론트→백엔드 연결RAILWAY_API_URL=https://redelivery-production.up.railway.app같은 변수명, 값만 https://<app>.fly.dev 로 교체
왜 백엔드만?
  • 프론트(Vercel) 는 안정적·빠름·무료 — 손댈 이유 없음
  • 오늘 incident 의 영향은 백엔드(Railway) 한정
  • 변경 범위 작음 → 롤백도 환경변수 한 줄

3. 조회 속도 영향

구간Railway SFOFly NRT
사용자(한국) → Vercel 프론트~10ms동일
Vercel 함수 → 백엔드60~80ms50~120ms (Vercel 함수 리전 의존)
백엔드 → Florens(홍콩) × 수십회~150ms × N~50ms × N ⬅ 핵심 개선
  • Playwright 가 Florens 와 주고받는 요청이 한 조회당 수십 건 → 누적 절감 의미 있음
  • 일반 조회 30~60초 기준 → 5~15초 단축 (약 15~25%)
  • 콜드 스타트 시 첫 요청 +5~10초 발생. 회피하려면 min_machines_running=1 (추가 비용 거의 없음)
  • 다른 임대사(TEXA/TRIT/GOLD/GESE) 의 호스팅 위치 알 수 없음 — 미국/유럽이면 NRT 가 약간 느릴 수 있으나 대부분 CDN 뒤라 큰 차이 없을 것

4. 코드/설정 변경 사항

#파일변경리스크
1 scraper/fly.toml (신규) 앱 이름, 리전 NRT, Dockerfile 빌더, 볼륨 mount /data, 머신 사양 낮음
2 scraper/Dockerfile 그대로 사용 가능. Fly 가 Dockerfile 인식. PORT env 우선순위 확인만 낮음
3 scraper/.dockerignore (신규) frontend/, __pycache__, *.pyc, .venv/, FLOR/, TEXA/, TRIT/, GOLD/, GESE/, *.html, *.md 제외 낮음 — 빌드 시간/이미지 크기 ↓
4 scraper/railway.json 삭제 또는 유지(무시됨) 없음
5 scraper/main.py CORS 이미 Vercel 도메인 허용 중일 가능성 — 사전 점검 후 미설정 시 추가 (fastapi.middleware.cors) 중간 — 누락 시 프론트 호출 실패
6 프론트 코드 변경 없음. Vercel 환경변수 RAILWAY_API_URL 값만 새 도메인으로 교체 없음

4.1 scraper/fly.toml 예시

app = "redelivery-scraper"          # 글로벌 고유, 가입 시 확정
primary_region = "nrt"

[build]
  dockerfile = "Dockerfile"

[env]
  PORT = "8080"
  FLOR_SESSION_DIR = "/data"
  PLAYWRIGHT_BROWSERS_PATH = "/ms-playwright"

[http_service]
  internal_port = 8080
  force_https = true
  auto_stop_machines = false        # 콜드 스타트 회피
  min_machines_running = 1

[[vm]]
  size = "shared-cpu-1x"
  memory = "1gb"

[mounts]
  source = "data"
  destination = "/data"

4.3 CORS 확인 포인트

scraper/main.py 에 CORSMiddleware 설정이 있는지 확인. 허용 origin 에 다음 포함 여부:

  • Vercel 운영 도메인 (예: https://<project>.vercel.app)
  • Vercel preview 도메인 (정규식 https://.*-<team>\.vercel\.app)
  • 로컬 dev (http://localhost:3000)

누락 시 추가 PR 필요 — Fly 이전 직전에 점검.

5. 데이터 이전 — /data 볼륨

대상 파일 (49MB):

  • credentials.json — 자격증명 (HA_FLOR_ID 등)
  • email_templates.json — 메일 양식
  • flor_session_HA.json, flor_session_SK.json — Playwright storage_state

5.1 이전 방법 — 두 가지

방식장점단점
① 임시 다운로드 endpoint 한 번에 zip — 빠름. 사용자 토큰으로 보호 가능 endpoint 추가 코드 필요. 작업 후 즉시 제거
② Railway SSH 직접 복사 추가 코드 없음. railway ssh + cat > local 각 파일 개별. 인코딩 주의 (바이너리 없으니 OK)

권장: 방식 ② — 4 개 파일이라 수동 복사가 더 빠르고 깔끔.

5.2 Fly 볼륨에 업로드

fly ssh sftp shell -a redelivery-scraper
> put credentials.json /data/credentials.json
> put email_templates.json /data/email_templates.json
> put flor_session_HA.json /data/flor_session_HA.json
> put flor_session_SK.json /data/flor_session_SK.json
세션 파일 이전 실패 시 Florens 슬라이더 캡차 1회 재로그인 발생. 서비스 중단 X, 사용자 시각엔 첫 조회만 ~10초 느림. 큰 문제 아님.

6. 환경변수 이전

Railway 의 Variables 탭 키 목록을 그대로 Fly secrets 로 옮김:

# Railway Variables 탭에서 키 목록 복사 → 한 번에:
fly secrets set \
  SK_FLOR_ID=xxx SK_FLOR_PW=xxx \
  HA_FLOR_ID=xxx HA_FLOR_PW=xxx \
  SK_TEXA_ID=xxx SK_TEXA_PW=xxx \
  HA_TEXA_ID=xxx HA_TEXA_PW=xxx \
  # ... TRIT/GOLD/GESE 등
  CREDENTIALS_ENCRYPTION_KEY=xxx \
  --app redelivery-scraper

주의:

  • FLOR_SESSION_DIR=/data — fly.toml [env] 에 이미 있어 별도 secret 불필요
  • PLAYWRIGHT_BROWSERS_PATH — Dockerfile 의 playwright install 기본 경로 확인 후 일치 여부 결정
  • PORT — Fly 는 8080 권장

7. 절차 (전체 순서)

S0 사전 점검 (15분)

  1. Fly.io 가입 / CLI 설치 — Windows: iwr https://fly.io/install.ps1 | iex
  2. fly auth login
  3. 현 Railway Variables 키 목록 캡처 (스크린샷 또는 export)
  4. 현 scraper/main.py CORS 설정 확인 → 필요 시 Vercel 도메인 추가 PR 미리
  5. 현 /data 파일 로컬 백업 (railway ssh 또는 임시 endpoint)

S1 Fly 앱/볼륨 생성 (20분)

  1. cd scraper && fly launch --no-deploy — 인터랙티브로 앱 이름/리전(nrt)/머신 사양 선택
  2. 생성된 fly.toml 을 §4.1 예시 기준으로 수정 (mount 추가, min_machines 등)
  3. fly volumes create data --size 1 --region nrt
  4. scraper/.dockerignore 신규 생성
  5. 커밋 (브랜치 migrate/fly)

S2 시크릿 + 첫 배포 (20분)

  1. fly secrets set ... 일괄 (Railway Variables 와 동일하게)
  2. fly deploy — 첫 배포
  3. fly logs 로 시작 확인 — Application startup complete 보일 때까지
  4. curl https://<app>.fly.dev/ 헬스체크

S3 데이터 + 검증 (20분)

  1. Railway /data 4개 파일 로컬 다운로드 (S0 에서 했으면 재사용)
  2. fly ssh sftp shell 로 /data 업로드
  3. fly machine restart — 세션 파일 재로드
  4. Fly 도메인 직접 curl -X POST /query ... 로 1건 FLOR 조회 테스트
  5. 로그에서 marker=..., storage_state 복원 시도, existing session valid 확인

S4 컷오버 (5분)

  1. Vercel 대시보드 → 프로젝트 → Settings → Environment Variables
  2. RAILWAY_API_URL 값을 https://<app>.fly.dev 로 교체 (이름 그대로 유지)
  3. Vercel Redeploy 또는 다음 배포에 자동 반영
  4. 운영 도메인에서 FLOR 조회 1회 → 결과 확인

검증 1주일 병행 운영

  1. Railway 백엔드 1주일 유지 (즉시 삭제 X)
  2. 매일 1회 운영 모니터링 — 콜드 스타트, 오류, Florens 응답성
  3. 1주일 후 Railway 서비스 삭제 + 볼륨 삭제
  4. scraper/railway.json 제거 커밋 (선택)

8. 롤백 계획

컷오버 후 Fly 측 문제 발생 시 5분 안에 원복 가능:

  1. Vercel 환경변수 RAILWAY_API_URL 을 https://redelivery-production.up.railway.app 로 되돌림
  2. Vercel Redeploy 또는 다음 요청에 자동 반영
  3. Railway 백엔드는 1주일 유지 중이라 즉시 응답

이게 가능한 이유: 코드 변경 자체가 Fly 측 추가만(fly.toml, .dockerignore) — Railway 측 코드는 손대지 않음. 두 인프라가 같은 commit 으로 양쪽에서 동시 가동 가능.
