# 프로젝트 규칙

## 협업 규칙

- **코딩 시작 전 반드시 협의**: 무엇을 만들지, 어떻게 만들지 명확히 합의가 된 후에만 구현을 시작한다.
- 방향이 불분명하거나 합의되지 않은 사항은 구현 전에 먼저 질문한다.
- 합의된 내용을 변경해야 할 경우 먼저 알리고 승인을 받는다.
- **개발 전 계획 필수**: 코딩 시작 전에 반드시 개발 계획을 `.md` 파일로 작성하고 제출한다.
- 계획 검토 및 협의가 완료된 후에만 코딩을 시작한다.
- 계획 내용: 목표, 기술 스택, 폴더 구조, 주요 기능, 작업 순서 등을 포함한다.

## 프로젝트 구조 규칙

- **임대사별 독립 개발**: 각 임대사(리스사)는 웹 접근 방식이 다르므로 임대사별로 개별 프로그램을 제작한다.
- **임대사별 폴더 분리**: 임대사마다 독립된 폴더로 관리한다.
- **순차 개발**: 한 임대사씩 차근차근 개발하며, 한 임대사가 완성된 후 다음 임대사로 진행한다.
- 각 임대사 폴더는 해당 임대사의 웹 접근 방식, 로직, 설정을 독립적으로 포함한다.

## 임대사(리스사) 문서 폴더 규칙

- **각 임대사별 문서 폴더 필수**: 임대사마다 루트에 임대사 코드명 폴더를 둔다 (예: `TEXA/`, `TRIT/`, `GOLD/`, `FLOR/`).
- 임대사 폴더 안에는 **문서/규칙만** 보관한다. 다음 두 파일을 기본으로 둔다:
  - `CLAUDE.md` — 해당 임대사 전용 개발 규칙
  - `ANALYSIS.md` — 해당 임대사 사이트 분석 결과
- **실행 코드는 임대사 폴더에 두지 않는다.** 자동화 코드는 `scraper/scrapers/<lessor>.py`, 공통 베이스는 `scraper/scrapers/base.py` 에 위치한다.
- 임대사별 `CLAUDE.md`에는 다음을 포함한다:
  - 임대사명, 코드, 사이트 URL, 접근 방식
  - 자격증명 환경변수 키 규칙 (선사별 prefix 포함)
  - 분석 문서 / 스크래퍼 / 공통 베이스의 위치 매핑
  - 임대사 고유의 주의사항
- 임대사별 규칙이 루트 규칙과 충돌할 때는 루트 규칙이 우선한다.
- 새 임대사 추가 시 동일 양식으로 폴더와 `CLAUDE.md`를 함께 생성한다.

## 선사 접근 통합 원칙

- **선사(장금상선/흥아라인)는 폴더로 분리하지 않는다.** 동일한 사용자가 한 화면에서 양사 데이터를 함께 다룬다.
- 선사 구분은 **결과 데이터의 `company` 키(`"SK"` / `"HA"`)** 로만 표시한다.
- 자격증명은 선사 prefix로 두 벌 저장한다: `SK_<LESSOR>_ID/PW`, `HA_<LESSOR>_ID/PW`.
- 사이트 분석은 임대사별 1회만 작성하고, 선사 간 차이점(계정, 도시 기본값, UI 차이 등)은 해당 `ANALYSIS.md` 끝에 별도 섹션으로 메모한다.
