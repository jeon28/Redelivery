# TRIT 개발 규칙

> 임대사: Triton
> 코드: `TRIT`
> 루트 `../CLAUDE.md` 규칙을 먼저 따르고, 본 문서의 규칙을 추가로 적용합니다.

## 1. 기본 정보

| 항목 | 값 |
|------|-----|
| 임대사명 | Triton |
| 코드 | `TRIT` |
| 로그인 URL | _(라이브 시연 시 기록)_ |
| 포털 URL | _(라이브 시연 시 기록)_ |
| 접근 방식 | 웹 자동화 (Playwright) |

## 2. 자격증명 키 규칙

| 선사 | ID 키 | PW 키 |
|------|-------|-------|
| 장금상선 (SK) | `SK_TRIT_ID` | `SK_TRIT_PW` |
| 흥아라인 (HA) | `HA_TRIT_ID` | `HA_TRIT_PW` |

- 선사 분기는 `company` 파라미터(`"SK"` / `"HA"`)로 처리하며, 스크래퍼가 해당 prefix의 환경변수를 로드한다.
- 본 문서 및 `ANALYSIS.md`에 실제 ID/PW를 절대 기재하지 않는다. 키 이름만 표기한다.

## 3. 위치 매핑

| 항목 | 경로 |
|------|------|
| 사이트 분석 문서 | `./ANALYSIS.md` _(라이브 시연 후 작성 예정)_ |
| 스크래퍼 구현 | `../scraper/scrapers/trit.py` _(분석/합의 후 구현)_ |
| 공통 베이스 클래스 | `../scraper/scrapers/base.py` |
| 자격증명 환경변수 | `../scraper/.env` |

## 4. 작업 규칙

- **분석 → 합의 → 구현 → 테스트** 순으로 진행하며, 각 단계 사이에 사용자 승인을 받는다.
- 사이트 분석은 사용자 라이브 시연을 기반으로 수행하며, 결과를 `ANALYSIS.md`에 정리한다.
- 사이트 UI/플로우가 변경되면 **먼저 `ANALYSIS.md`를 갱신** 한 후 `trit.py`를 수정한다.
- 선사 간 차이점은 `ANALYSIS.md` 끝의 "선사 차이점" 섹션에 기록한다.

## 5. 주의사항

- **예약/Book 버튼 = 실 발급 가능성**: 동일 컨테이너 중복 실행 방지 로직 필수 (Triton 사이트의 정확한 동작은 라이브 시연으로 확인).
- 분석 시 다음 항목을 최소한으로 캡처한다:
  - 로그인 URL 및 폼 필드명
  - 반납 조회 메뉴 진입 경로
  - 입력 필드 (지역, 컨테이너 번호, 날짜 등)
  - 결과 화면 구조 (가능/불가, 반납번호, 반납지, 유효기간)
  - 반납번호 발급 트리거(버튼) 및 발급 후 상세 화면
