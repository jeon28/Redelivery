TEXA — 최초 반납신청 시 반납번호 회수 실패 버그 수정 계획

대상 파일: scraper/scrapers/texa.py · 작성일 2026-06-01 · 상태: 승인 대기

1. 증상

  • 처음 반납신청하는 컨테이너 조회 시 "임대사 시스템에 해당 컨테이너 정보 없음" (불가) 로 표시됨. (Image #1)
  • 그러나 임대사 사이트에는 정상 부킹이 생성됨 — New Booking - TKF6D00 is created successfully! (Image #2)
  • 같은 컨테이너를 다시 조회하면 정상적으로 가능 + 반납번호(TKF6C91) 가 나옴. (Image #3)

2. 근본 원인

_collect_and_visit_bookings() 가 Book 직후 발급된 반납번호 링크를 옛 prefix 문자열로 하드코딩 필터하고 있음:

scraper/scrapers/texa.py:830
bk_links = frame.locator("a").filter(has_text="TKE")
  • 현재 실제 반납번호는 TKF6D00, TKF6C91 처럼 TKF 로 시작 → "TKE" 필터에 0개 매칭.
  • 결과적으로 results 가 하나도 채워지지 않고, query() 말미 폴백 루프가 "정보 없음 / 불가" 로 마감.
  • 반납번호의 3번째 글자는 시기에 따라 굴러감(5월 TKE… → 6월 TKF…). prefix 하드코딩 자체가 취약.
Preview "can be booked"→ Book 클릭(실 예약 생성)→ TKE 링크 탐색 실패(0개)→ 결과 미기입 → 불가

3. 수정 방안

방안 B 권장 — Book 후 재-Preview → 이미 검증된 경로 재사용

Book 으로 예약이 생성되면, 동일 컨테이너로 Preview를 한 번 더 실행한다. 그러면 해당 컨테이너들은 "are booked" 상태가 되고, 이미 정상 동작이 검증된 _parse_booked_table() 가 반납번호·반납지·Over Caps·유효기간을 정확히 매핑한다. (Image #3 이 바로 이 경로의 출력)

  • 장점: 실제로 동작하는 경로를 재사용 → 링크 레이아웃/ prefix 변화에 무관. 컨테이너↔반납번호 매핑이 정확.
  • 중복예약 위험 없음: Preview 는 조회 전용이며, 이미 예약된 상태에서는 Book 버튼이 없고 "Can be deleted" 패널만 노출됨.
  • Book 직후 success 메시지(New Booking - {ref} is created successfully!)에서 발급 ref 를 로그로 남겨 교차검증.

방안 A 대안/보강 — 링크 필터를 정규식으로 교정

has_text="TKE" 를 has_text=re.compile(r"^TK[A-Z0-9]") 같은 패턴으로 바꿔 TKF 등 미래 prefix까지 포착.

  • 단점: Book 직후 화면이 실제로 "링크 목록"인지 불확실(Image #2 는 success 메시지 + 빈 폼만 보임). 이 화면 가정이 틀리면 또 실패.
제안: 방안 B 를 메인 경로로 구현하고, 방안 A 의 정규식 교정도 함께 반영해 혹시 링크형 화면이 나오는 계정/상황에서도 안전하도록 이중 안전장치를 둔다.

4. 작업 순서

  1. ANALYSIS.md 갱신: 반납번호 prefix가 시기별로 변동(TKE→TKF)함을 명시하고, "Book 후 재-Preview로 회수" 흐름 기록.
  2. texa.py Case 2 ("can be booked") 분기 수정:
    • Book 클릭 → success 메시지에서 발급 ref 추출(로그/검증용).
    • 동일 컨테이너로 재-Preview → _parse_booked_table() 호출로 결과 확정.
    • _collect_and_visit_bookings 의 "TKE" → 정규식으로 교정(보강).
  3. 검증: 신규 컨테이너 1건으로 조회 → 첫 조회에서 바로 "가능 + 반납번호" 표시되는지 확인. ⚠ 실 예약이 생성되므로, 검증 후 취소(cancel) 플로우로 정리.
