# TEXA 조회 실패 가시화 계획

## 목표

현재 `texa.py`는 모든 실패 경로에서 사용자에게 똑같이 `reason: "조회 실패"` 또는
`"조회 실패 (사유 미상)"`만 노출한다. 그 결과:

- 사용자는 왜 실패했는지 모르고,
- 개발자도 추측만 가능하다 (로그인? 사이트 다운? 컨테이너 매칭 실패?).

본 작업의 목적은 **새 기능 추가가 아니라**, 실패가 발생한 **단계와 원인을 결과
`reason` 문자열로 명확히 구분**하는 것이다. 이를 통해:

1. 사용자는 "이건 다시 시도하면 되겠다" / "이건 컨테이너 번호 잘못 입력" 같은
   판단을 즉시 할 수 있다.
2. 개발자는 빈도가 가장 높은 실패 단계를 데이터로 확인해, 다음 개선
   (세션 재사용 / 고정 sleep 제거 등)의 우선순위를 정한다.

## 변경 범위

`scraper/scrapers/texa.py` **단일 파일만 수정**. 다른 임대사·프론트엔드는
건드리지 않는다. `ContainerResult` 스키마도 변경 없음 (기존 `reason` 필드 재사용).

## 구체적 변경

### 1. 초기 reason 제거

현재 `query()` 시작 시 모든 컨테이너에 `reason: "조회 실패"`를 미리 박아둔다.
이게 모든 후속 단계의 실패를 같은 메시지로 덮어버린다.

→ 초기 `reason: None`으로 시작. 단계별로 실패 사유를 명시적으로 기록한다.

### 2. 단계별 실패 사유 부여

`query()` try 블록을 단계로 나누고, 각 단계 실패 시 **모든 컨테이너의** reason을
해당 단계 메시지로 채운 뒤 early return.

| 단계 | 실패 시 reason |
|------|----------------|
| `_navigate_to_redelivery()` | `"TEXA 포털 메뉴 진입 실패 (사이트 응답 지연 가능)"` |
| `_frame_containing("Country")` None | `"반납 신청 화면 로드 실패"` |
| Country/City select 예외 | `"지역 선택 실패 (사이트 UI 변경 가능)"` |
| Preview 클릭 후 3개 케이스 어디에도 미매칭 | `"Preview 응답 해석 불가 (세션 만료 또는 사이트 변경 의심)"` |
| 그 외 일반 예외 | `f"TEXA 조회 오류: {type(exc).__name__}"` (예: `TimeoutError`) |

### 3. 컨테이너 단위 매칭 실패 구분

- `cannot be redelivered` / `are booked` 테이블에 등장했지만 결과 dict 키와
  매칭이 안 되는 경우: 그 컨테이너만 `"컨테이너 번호 확인 필요 (체크디짓 누락 가능)"`
- 세 테이블 어디에도 나타나지 않은 경우: `"임대사 시스템에 해당 컨테이너 정보 없음"`

후자는 `_parse_booked_table`의 매칭 로직을 `startswith` 양방향 매칭으로 보강
(현재 `cannot` 경로에만 있는 로직을 booked 경로에도 적용).

### 4. 로그인 실패는 이미 base에서 RuntimeError 발생

`base.py:66`에서 `f"{self.lessor} 로그인 실패"` 던짐 → `routers/query.py` 까지
올라가서 FastAPI 500이 됨 → 프론트의 `data.error`로 잡혀 화면 상단에
`"TEXA 로그인 실패"`로 표시. 별도 처리 불필요.

### 5. 로깅 보강 (최소)

각 단계 진입/탈출에 이미 `logger.info`가 있다. 단계 실패 분기에 어떤 reason을
박았는지 한 줄 추가 정도. 새 핸들러나 외부 전송은 없음.

## 비변경

- 세션 재사용 (storage_state) 도입 — **이번 PR 범위 아님**. 본 가시화로 로그인 단계
  실패 비중을 먼저 확인한 뒤 별도 작업.
- 고정 sleep → wait_for_* 전환 — 동일 사유로 별도 작업.
- `over_caps` / `close_date` 등 기존 정상 경로 로직 — 손대지 않는다.

## 검증

1. 로컬에서 TEXA 정상 조회 1회 — 기존 동작 회귀 없는지 확인 (booked 케이스).
2. 임의로 잘못된 컨테이너 번호 1건 섞어 조회 — `"임대사 시스템에 해당 컨테이너 정보 없음"`
   메시지 표시 확인.
3. 자격증명을 일부러 틀려서 1회 — 화면 상단 `"TEXA 로그인 실패"` (data.error) 확인.

## 작업 순서

1. 본 계획 사용자 승인
2. `texa.py` 수정 (단일 파일, ~30~50줄 변경 예상)
3. 로컬 테스트 (위 검증 1·2)
4. 커밋 — 메시지 예: `TEXA: 조회 실패 사유 단계별 가시화`
